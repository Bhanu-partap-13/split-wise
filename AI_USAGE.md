## Key Prompts Used During Development

### Prompt 1 — Initial CSV Analysis
> *"Analyze this CSV file and identify every data quality problem. For each problem, tell me: what the issue is, which rows are affected, and what action should be taken during import."*

**Purpose**: Before writing any code, Gemini analyzed the `expenses_export.csv` to enumerate all anomaly types. This drove the design of the 10-check validation engine in `csvValidator.ts`.

**Output used**: The 10 anomaly categories (DUPLICATE_EXPENSE, IS_SETTLEMENT, MISSING_PAYER, etc.) were directly derived from Gemini's analysis.

---

### Prompt 2 — Database Schema Design
> *"Design a relational database schema for a Splitwise-like app. It must support: multiple groups, users, equal/unequal/percentage/share splits, CSV imports with external (non-registered) payers, settlements, real-time chat per expense, and import audit logs."*

**Purpose**: Schema first-pass for `convex/schema.ts`.

**Output used**: The core table structure (`users`, `groups`, `groupMembers`, `expenses`, `expenseSplits`, `settlements`, `messages`, `importBatches`) came from this prompt. The `paidByName` fallback field and `importBatchId` on expenses were Gemini suggestions adopted verbatim.

---

### Prompt 3 — Debt Simplification Algorithm
> *"Write a TypeScript function that takes a list of expense objects (each with paidById and splits[]) and settlement objects, builds a net balance map, and then runs a greedy debt simplification to minimize the number of transactions needed to settle all debts."*

**Purpose**: Generate `src/lib/balances.ts`.

**Output used**: The overall structure was adopted. Corrections were required (see Cases section below).

---

### Prompt 4 — AI System Prompt Engineering
> *"Write a system prompt for a Gemini chat model that will be embedded in a group expense tracking app. The model should only answer questions about the specific group's data (expenses, settlements, balances). Include the full expense list, settlement list, and simplified balance list as context. It should use member names, not user IDs, in all responses."*

**Purpose**: The system instruction in `src/app/api/ai-chat/route.ts`.

**Output used**: The full system prompt structure with sections (ALL EXPENSES, RECORDED SETTLEMENTS, OUTSTANDING BALANCES, RULES) was generated by Gemini and then refined.

---

### Prompt 5 — CSV Column Alias Map
> *"Give me a comprehensive list of all common column header aliases people use for: date, description, paid_by, amount, currency, split_type, split_with, split_details, and notes in expense tracking CSV exports from apps like Splitwise, Excel, and Google Sheets."*

**Purpose**: The `COLUMN_ALIASES` map in `csvValidator.ts` that allows flexible column naming.

**Output used**: The alias list was almost entirely AI-generated and then extended with a few custom cases.

## Three Concrete Cases Where AI Was Wrong

### Case 1: Wrong Debt Simplification — Creditor/Debtor Matching

**What AI generated:**
```typescript
// AI version — matched smallest creditor to smallest debtor
while (creditors.length > 0 && debtors.length > 0) {
  const creditor = creditors.shift(); // smallest first
  const debtor = debtors.shift();
  // ...
}
```

**The problem**: Sorting from smallest to largest and popping from the front produced more transactions than necessary for common cases. The optimal greedy algorithm matches the *largest* creditor against the *largest* debtor in each step to clear the maximum debt per transaction.

**How I caught it**: Manual trace with a 3-person example (A owes B ₹100, B owes C ₹100, A owes C ₹50). The AI version produced 3 transactions; the correct greedy algorithm produces 2.

**What I changed**: Switched to sorting descending and using an index pointer to always pick the largest outstanding creditor/debtor pair first:
```typescript
// Fixed version
creditors.sort((a, b) => b.amount - a.amount);
debtors.sort((a, b) => b.amount - a.amount);
```

### Case 2: AI Hallucinated a Convex API That Does Not Exist

**What AI generated:**
```typescript
// AI tried to use ctx.db.upsert() which does not exist in Convex
await ctx.db.upsert("groupMembers", { groupId, userId }, { role: "member", joinedAt: Date.now() });
```

**The problem**: Convex does not have a `.upsert()` method. The correct pattern is to query for an existing document first, then conditionally insert.

**How I caught it**: The Convex TypeScript compiler threw a type error immediately: `Property 'upsert' does not exist on type 'DatabaseWriter'`.

**What I changed**: Replaced with the standard Convex pattern used throughout the codebase:
```typescript
// Fixed version (now in convex/groups.ts addMember)
const existing = await ctx.db
  .query("groupMembers")
  .withIndex("by_group_and_user", (q) => q.eq("groupId", args.groupId).eq("userId", args.userId))
  .first();
if (existing) return; // already a member
await ctx.db.insert("groupMembers", { ...args, role: "member", joinedAt: Date.now() });
```

### Case 3: AI Sent Full Expense Data to the AI Chat Client-side

**What AI originally suggested:**
```typescript
// AI suggested fetching and passing raw expense objects to Gemini from the frontend
const expenses = await fetchAllExpenses(groupId); // thousands of rows possible
const prompt = `Here are all expenses: ${JSON.stringify(expenses)}`;
```

**The problem**: This approach would:
1. Expose all raw database IDs and internal fields to the browser's Network tab
2. Risk exceeding Gemini's context window for large groups (50+ expenses)
3. Send unfiltered user data (including notes) to a third-party API without truncation

**How I caught it**: During code review, I noticed the entire expense array (including `_id` fields, internal Convex metadata) was being serialized and sent directly to the LLM. This is a privacy/security concern.

**What I changed**:
1. Moved all LLM calls to the **server-side** (`/api/ai-chat/route.ts`) — the browser only sends the user's message text
2. Added a **cap of 50 expenses** in the context window (`expenses.slice(0, 50)`)
3. Added a **member authorization check** server-side — the API rejects requests from users not in the group
4. Resolved all member names to human-readable strings server-side before building the prompt, so no internal IDs are sent to Gemini
